NTP 时间同步模块详解

从零理解：UDP 协议、NTP 报文格式、时间戳转换、ESP8266 二进制数据发送

一、模块概述

NTP（Network Time Protocol）模块负责通过互联网获取准确的 UTC 时间。项目中使用它让 OLED 时间页显示真实的时钟，而不是芯片内部的自由计数。

1.1 为什么要用 NTP 而不是 HTTP？

对比NTPHTTP 获取时间
协议UDP（无连接）TCP（三次握手）
数据量48 字节请求 + 48 字节响应几百字节 HTTP 头 + 响应体
交互次数1 次交换TCP 握手 3 次 + HTTP 请求 1 次 + 响应
精度毫秒级（带小数部分）秒级（HTTP Date 头）
是否需要 API Key不需要取决于此服务

二、NTP 协议原理

2.1 NTP 是什么？

NTP（Network Time Protocol）是互联网上广泛使用的时间同步协议。它通过 UDP 协议（端口 123）与时间服务器通信，获取精确的 UTC 时间。

核心思路很简单：

  • 客户端发送 48 字节的请求包
  • 服务器返回 48 字节的响应包，其中包含精确的时间戳
  • 客户端提取时间戳，转换为本地时间

2.2 NTP vs Unix 时间戳

NTP 时间戳Unix 时间戳
起点 (Epoch)1900-01-01 00:00:001970-01-01 00:00:00
单位秒（整数）+ 1/2^32 秒（小数）秒
编码大端 4 字节（秒部分）小端/大端根据平台

转换公式：Unix 时间戳 = NTP 时间戳 - 2208988800

这个数字是 1900-01-01 到 1970-01-01 之间的总秒数：70 年 × 365.25 天 × 86400 秒 ≈ 2208988800

2.3 为什么选择 pool.ntp.org？

  • 免费：无需注册、无需 API Key
  • 分布式：负载均衡，全球多个服务器
  • 低延迟：UDP 直连，无 TCP 握手开销
  • 标准协议：所有 NTP 客户端都兼容

三、ntp_get_time 逐行拆解

这是整个模块最核心的函数，它通过 ESP8266 的 AT 命令实现 UDP 通信，完成 NTP 协议交互。

int ntp_get_time(const char* server, uint32_t* timestamp)
{
    char cmd[128];
    uint8_t resp_buf[256];      // 原始响应缓冲区
    uint8_t ntp_req[48];        // NTP 请求包
    int ret;

    /* 步骤1: 参数检查 */
    if (server == NULL || timestamp == NULL) return -1;

    /* 步骤2: 清空环形缓冲区，避免旧数据干扰 */
    esp8266_rx_flush();

    /* ⭐ 步骤3: 建立 UDP 连接 */
    snprintf(cmd, sizeof(cmd),
        "AT+CIPSTART=\"UDP\",\"%s\",%d,%d,0\r\n",
        server, NTP_SERVER_PORT, NTP_SERVER_PORT);
    //                          123       123
    //                         远程端口   本地端口
    esp8266_send_at(cmd);
    ret = esp8266_wait_str("OK", 5000);
    if (ret != 0) {
        esp8266_send_at("AT+CIPCLOSE\r\n");
        return -2;  // UDP 连接失败
    }

    /* 步骤4: 告诉 ESP8266 要发送 48 字节数据 */
    snprintf(cmd, sizeof(cmd), "AT+CIPSEND=%d\r\n", NTP_PACKET_SIZE);
    esp8266_send_at(cmd);
    ret = esp8266_wait_str(">", 5000);
    if (ret != 0) {
        esp8266_send_at("AT+CIPCLOSE\r\n");
        return -3;  // 没收到 > 提示符
    }

    /* ⭐ 步骤5: 发送 NTP 请求包（二进制数据！）*/
    build_ntp_packet(ntp_req);               // 构造 48 字节请求
    esp8266_send_raw((const char*)ntp_req, NTP_PACKET_SIZE);
    //               ⚠ 用 send_raw 而不是 send_at！

    /* ⭐ 步骤6: 接收 NTP 响应（10 秒超时）*/
    {
        uint32_t start = HAL_GetTick();
        uint32_t idx = 0;
        while (HAL_GetTick() - start < NTP_RESP_TIMEOUT)
        {
            char batch[32];
            int n = esp8266_rx_read(batch, sizeof(batch));
            for (int i = 0; i < n && idx < sizeof(resp_buf) - 1; i++)
                resp_buf[idx++] = batch[i];
            if (n == 0) osDelay(1);
        }
    }

    /* 步骤7: 关闭 UDP 连接 */
    esp8266_send_at("AT+CIPCLOSE\r\n");

    if (idx == 0) return -4;  // 没收到任何数据

    /* ⭐ 步骤8: 在原始响应中解析 +IPD,48: 帧头 */
    // 在 resp_buf 中查找 "+IPD," 字符串
    char* ipd = NULL;
    for (uint32_t i = 0; i + 5 < idx; i++)
    {
        if (resp_buf[i] == '+' && resp_buf[i+1] == 'I' &&
            resp_buf[i+2] == 'P' && resp_buf[i+3] == 'D' &&
            resp_buf[i+4] == ',')
        {
            ipd = (char*)&resp_buf[i];
            break;
        }
    }
    if (ipd == NULL) return -5;  // 没找到 +IPD 头

    /* 步骤9: 跳过 "+IPD,N:" 帧头，定位到数据体 */
    char* p = ipd + 5;           // 跳过 "+IPD,"
    while (*p >= '0' && *p <= '9') p++;  // 跳过数字长度
    if (*p != ':') return -6;   // 格式错误
    p++;  // 跳过冒号

    /* 步骤10: 提取 NTP 时间戳（响应字节 40-43）*/
    if ((uint32_t)(p + 43 - (char*)resp_buf) >= idx)
        return -7;  // 响应太短

    // 大端 4 字节 → uint32_t
    uint32_t ntp_time = ((uint32_t)(uint8_t)p[40] << 24) |
                        ((uint32_t)(uint8_t)p[41] << 16) |
                        ((uint32_t)(uint8_t)p[42] << 8) |
                        (uint32_t)(uint8_t)p[43];

    /* 步骤11: NTP → Unix 转换 */
    if (ntp_time < NTP_EPOCH_OFFSET) return -8;
    *timestamp = ntp_time - NTP_EPOCH_OFFSET;  // -2208988800

    return 0;  // 成功
}

3.1 AT 命令交互全流程

1 AT+CIPSTART="UDP","pool.ntp.org",123,123,0
→ 等待 "OK"（5 秒超时）
2 AT+CIPSEND=48
→ 等待 ">"（5 秒超时）
3 发送 48 字节 NTP 请求包（二进制）
→ ESP8266 转发到 pool.ntp.org:123
4 接收响应：+IPD,48:（48 字节 NTP 响应）
→ 解析 +IPD 帧头 → 提取 byte 40-43
5 AT+CIPCLOSE

3.2 为什么用 send_raw 而不是 send_at？

关键问题：NTP 请求包包含 0x00 字节（48 字节中前 47 字节大部分是 0），而 esp8266_send_at 内部用 strlen 计算长度。strlen 遇到第一个 0x00 就返回 0，所以实际发出去的是空数据！ 
// ❌ 错误：strlen(ntp_req) 遇到 0x00 返回 0
esp8266_send_at((const char*)ntp_req);  // 实际发送 0 字节

// ✅ 正确：明确指定长度 48 字节
esp8266_send_raw((const char*)ntp_req, 48);

这也就是为什么要将 esp8266_send_raw 暴露为公共函数（原 send_cmd/send_data 都是 static 的）。

四、NTP 报文详解

4.1 构造 NTP 请求

static void build_ntp_packet(uint8_t* pkt)
{
    memset(pkt, 0, NTP_PACKET_SIZE);  // 全部清零
    pkt[0] = 0x1B;  // 设置首字节
}

NTP 请求包格式（48 字节）：

偏移大小字段值说明
01LI/VN/Mode0x1BVN=3 (版本3), Mode=3 (客户端)
11Stratum0x00请求包填 0
2-32Poll/Prec0x00 0x00请求包填 0
4-3936保留/根延时/...全 0保留字段
40-478Transmit Timestamp0 (原样发送)服务器会复制到 Originate Timestamp

4.2 解析 NTP 响应

服务器返回也是 48 字节。关键字段在字节 40-43（Transmit Timestamp 的整数秒部分）：

偏移大小说明
0-3940服务器状态信息（我们不需要解析）
40-434Transmit Timestamp 秒部分（大端）→ 这就是我们要的数据！
44-474小数部分（1/2^32 秒精度，我们不需要）

4 字节大端整数转 uint32_t 的代码：

uint32_t ntp_time = (p[40] << 24) | (p[41] << 16) |
                    (p[42] << 8)  |  p[43];

4.3 时间戳转换实例

假设服务器返回的 NTP 时间戳 = 3992345678（十六进制 0xEDE5B84E）

NTP 时间戳:  3992345678 秒（自 1900-01-01）
减去偏移:    -2208988800
Unix 时间戳:  1783356878 秒（自 1970-01-01）

换算为人类时间：1783356878 → 2026-06-19 18:14:38（UTC+8）

五、时间格式化 — ntp_format_time

5.1 代码

void ntp_format_time(char* buf, uint32_t timestamp)
{
    // 取一天的秒数（取模 86400）
    uint32_t sec = timestamp % 86400;

    // 计算时/分/秒
    uint32_t h = sec / 3600;            // 小时（0-23）
    uint32_t m = (sec % 3600) / 60;   // 分钟（0-59）
    uint32_t s = sec % 60;              // 秒钟（0-59）

    // 手动格式化 "HH:MM:SS"（不用 sprintf，节省栈空间）
    buf[0] = '0' + h / 10;     // 十位小时
    buf[1] = '0' + h % 10;     // 个位小时
    buf[2] = ':';
    buf[3] = '0' + m / 10;     // 十位分钟
    buf[4] = '0' + m % 10;     // 个位分钟
    buf[5] = ':';
    buf[6] = '0' + s / 10;     // 十位秒钟
    buf[7] = '0' + s % 10;     // 个位秒钟
    buf[8] = '\0';               // 结束符
}

5.2 为什么不用 sprintf？

// sprintf 版本（更简洁但更占栈）
sprintf(buf, "%02d:%02d:%02d", h, m, s);

// 手写版本（更安全，不引入 stdio 格式化的栈开销）
buf[0] = '0' + h/10; ...

手写版本的好处：

  • 不依赖 sprintf 的格式化解析器（减少代码体积）
  • 固定的 9 字节栈使用（sprintf 内部可能有几十字节的临时缓冲区）
  • 执行速度更快（无格式化解析开销）

5.3 UTC+8 时区处理

// 在 freertos.c WIFITask 中，NTP 同步后加 8 小时
system_time = ts + 8 * 3600;   // UTC → UTC+8（杭州时间）
time_synced = 1;

// TimeTask 中每秒递增
if (time_synced) system_time++;  // 秒递增
osDelay(1000);

时序：NTP 获取的是 UTC 时间，加 8 小时转为北京时间（UTC+8）。之后 TimeTask 每秒 system_time++ 维持时间更新。

六、关键问题与经验教训

6.1 Bug 汇总

#问题根因修复
Bug 1 NTP 首次同步必失败 TimeTask 的 NTP 和 WIFITask 的 HTTP GET 同时操作 USART1，AT 命令交错 使用事件组 NTP_ALLOWED_BIT，等 HTTP GET 完成后再开始 NTP
Bug 2 NTP 发送 0 字节 误用 esp8266_send_at（用 strlen 算长度），0x00 导致截断 改用 esp8266_send_raw 明确指定 48 字节
Bug 3 NTP 重试等待 15 秒太长 osDelay(15000) 让用户等待太久 改为 osDelay(5000)
Bug 4 页面 2 切换后黑屏 time_synced=0 时更新条件不满足 last_tick==0 时无条件绘制 "--:--:--"

6.2 Bug 1 详细分析：WIFITask 和 TimeTask 竞争 USART1

现象：串口输出显示 NTP 和 HTTP 的 AT 命令交错发送，ESP8266 返回 "busy s..."。
根因：两个任务同时向 USART1 写数据，ESP8266 收到乱码。 
// 时间轴线 (改前):
WIFITask:    |--- esp8266_init ---||--- WiFi连接 ---||--- HTTP GET ---------------||
TimeTask:                           |--- ntp_get_time ---|                           |
                                    ↑ 两个任务同时用 USART1 → AT 命令交错 → 都失败  ↑

// 时间轴线 (改后):
WIFITask:    |--- init ---|--- WiFi ---|-- HTP GET --|-- sleep 30min --|-- HTTP --...
TimeTask:                               wait bit0     |-- NTP ----------|
                                                      ↑ WIFITask 休眠时
                                                        TimeTask 独占 USART1

6.3 事件组同步的完整实现

// freertos.c
#define WIFI_CONNECTED_BIT     (1 << 0)  // bit 0: WiFi 硬件连接成功
#define NTP_ALLOWED_BIT        (1 << 1)  // bit 1: USART1 空闲, NTP 可开始

// WIFITask: WiFi 连上后设置 bit0
osEventFlagsSet(systemEventGroupHandle, WIFI_CONNECTED_BIT);
// ... 完成 HTTP GET 后设置 bit1
osEventFlagsSet(systemEventGroupHandle, NTP_ALLOWED_BIT);

// TimeTask: 分两步等待
osEventFlagsWait(systemEventGroupHandle, WIFI_CONNECTED_BIT, ...);  // 等WiFi
osEventFlagsWait(systemEventGroupHandle, NTP_ALLOWED_BIT, ...);    // 等HTTP完成
// 此时 WIFITask 已进入 osDelay(1800000)，USART1 空闲 → 安全 NTP

6.4 经验教训

教训1：RTOS 下共享外设必须互斥

一个 USART1 连接一个 ESP8266，任何时刻只能有一个任务发 AT 命令。FreeRTOS 下必须用同步原语协调访问。"两个任务同时"在 RTOS 中不是偶然，而是调度的必然。

教训2：事件组可以做分阶段同步

只用 WIFI_CONNECTED_BIT 不够——它只表示"WiFi 连接完成"，不表示 "USART1 空闲"。用两个位分别表示"就绪"和"允许"，可以实现更精细的流程控制。

教训3：二进制数据不能用字符串函数

凡是包含 0x00 的数据包，都不能用 strlen、strcpy 等 C 字符串函数处理。NTP 请求包就是典型的反例——48 字节中有 47 个是 0x00。
