ESP8266 WiFi 驱动模块详解

从零理解：USART1 中断接收、环形缓冲区、AT 命令解析、HTTP GET 数据清洗

一、模块概述

ESP8266 驱动是项目的"联网大脑"——它负责三件事：

  1. 初始化 WiFi 模块：设置 AT 模式、连接路由器
  2. 发起 HTTP 请求：向心知天气 API 发送 GET 请求并接收响应
  3. 提供底层 AT 命令收发接口：供 NTP 等其他模块使用

设计思想：公共接口（send_at/send_raw/wait_str）提供给 ntp.c 使用，而内部函数 esp8266_send_cmd/send_data/wait_response 用 static 隐藏，只在本文件可见。这样 NTP 模块不需要了解底层实现细节。

二、环形缓冲区（Ring Buffer）

2.1 为什么需要环形缓冲区？

ESP8266 发数据 (115200 baud) → USART1 硬件接收 → 中断触发 → HAL_UART_RxCpltCallback → 写入环形缓冲区 → WIFITask 从中读取

核心问题：中断服务程序（ISR）是"生产者"，收到一个字节就触发一次；任务（Task）是"消费者"，需要的时候才批量读取。如果 ISR 直接把字节塞给任务，会产生竞态条件。环形缓冲区就是中间的"快递驿站"——ISR 往里放，任务往外取，互不干扰。

2.2 代码逐行解释

#define ESP8266_RING_BUF_SIZE 512   // 缓冲区的最大容量（字节数）

static char esp8266_ring_buf[ESP8266_RING_BUF_SIZE];  // 实际的存储空间
static volatile uint16_t ring_head = 0;  // 写入位置（ISR 修改）
static uint16_t ring_tail = 0;             // 读取位置（任务修改）

ring_head 用 volatile 修饰，因为它在 ISR 中被修改，编译器需要每次都从内存读取，不能缓存到寄存器中。

（1）ISR 写入（生产者）

void esp8266_rx_callback(uint8_t data)
{
    // 计算下一个写入位置（取模实现循环）
    uint16_t next = (ring_head + 1) % ESP8266_RING_BUF_SIZE;

    // 检查缓冲区是否已满：如果 next 追上 tail 说明满了
    if (next != ring_tail)
    {
        esp8266_ring_buf[ring_head] = data;  // 存数据
        ring_head = next;                     // 移动写指针
    }
    // 如果满了, 直接丢弃数据（硬件上不可避免的丢包）
}

（2）任务读取（消费者）

int esp8266_rx_read(char *buf, uint32_t len)
{
    uint32_t i = 0;
    // 循环读取，最多读 len 个字节，或者直到缓冲区为空
    while (i < len && ring_tail != ring_head)
    {
        buf[i++] = esp8266_ring_buf[ring_tail];        // 取一个字节
        ring_tail = (ring_tail + 1) % ESP8266_RING_BUF_SIZE;  // 移动读指针
    }
    return i;  // 返回实际读取到的字节数
}

（3）清空缓冲区

void esp8266_rx_flush(void)
{
    ring_head = 0;  // 直接重置两个指针就相当于清空
    ring_tail = 0;  // 旧数据会被后续写入覆盖
}

2.3 环形缓冲区的工作原理图解

状态说明head==tail?
空head == tail，没有数据可读✓
有数据head != tail，有 (head - tail) 个字节等待读取✗
满head+1 == tail（取模后），不能写入，否则覆盖未读数据✗

注意：缓冲区最多存 511 字节（不是 512），因为要保留一个"空位"来区分空和满。全满时 head 的下一格就是 tail，这时不能再写。

新手常见疑问：为什么不用队列（Queue）？因为 FreeRTOS 队列涉及任务上下文切换，不适合在中断中用。环形缓冲区是纯内存操作，ISR 中写入纳秒级完成。

三、中断接收完整流程

3.1 启动中断接收

// esp8266.c:119（在 esp8266_init 中调用）
HAL_UART_Receive_IT(&huart1, &esp8266_rx_byte, 1);

HAL_UART_Receive_IT 的作用：使能 USART1 的 RXNEIE 位（接收缓冲区非空中断），告诉 UART 硬件"收到一个字节后触发中断"。

但这只是"外设级使能"，还需要"系统级使能"——NVIC（嵌套向量中断控制器）：

// usart.c:117-118（在 HAL_UART_MspInit 中配置）
HAL_NVIC_SetPriority(USART1_IRQn, 6, 0);   // 设置中断优先级
HAL_NVIC_EnableIRQ(USART1_IRQn);             // 使能 NVIC 中断

3.2 中断响应链

1 USART1 收到一个字节 → 硬件自动将数据放入 USART1->DR 寄存器
2 CPU 跳转到 USART1_IRQHandler → 在 stm32f1xx_it.c:178
3 HAL_UART_IRQHandler 判断中断类型 → 发现是 RXNE 中断
4 HAL 库调用用户回调 HAL_UART_RxCpltCallback 
// esp8266.c:13-20
void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart)
{
    if (huart->Instance == USART1)
    {
        esp8266_rx_callback(esp8266_rx_byte);  // 存入环形缓冲区
        HAL_UART_Receive_IT(&huart1, &esp8266_rx_byte, 1);  // 重启接收！
    }
}

关键：最后一行必须重启接收。HAL 库的接收中断是一次性的——处理完一个字节后 RXNEIE 位会被清除。如果没有 HAL_UART_Receive_IT 再次使能，后续字节将不再触发中断。

3.3 完整时序图

ESP8266 发 'A' (0x41) → USART1 DR 寄存器收到 → 硬件置 RXNE 标志 → NVIC 响应中断 → USART1_IRQHandler → HAL_UART_IRQHandler → RxCpltCallback → ring_buf[head]=0x41 → 重启接收

上述流程在 115200 baud 下每 ~87 μs 发生一次（一个字节的传输时间：10 bits / 115200 = 86.8 μs）。

四、AT 命令收发

4.1 发送 AT 命令

static void esp8266_send_cmd(const char* cmd)
{
    // 直接调用 HAL 库的 UART 发送函数
    HAL_UART_Transmit(&huart1, (uint8_t*)cmd, strlen(cmd), 1000);
}

HAL_UART_Transmit 是阻塞发送，第三个参数是超时时间（ms）。在 FreeRTOS 环境下，阻塞时调度器可以切换到其他任务运行。

4.2 等待响应（核心逻辑）

static int esp8266_wait_response(const char* expected_str, uint32_t timeout)
{
    static char line_buf[256];   // 累加缓冲区，static 保存上次状态
    static uint16_t idx = 0;
    uint32_t start = HAL_GetTick();

    while (HAL_GetTick() - start < timeout)
    {
        if (esp8266_rx_available())
        {
            char buf[2];
            esp8266_rx_read(buf, 1);   // 读 1 个字节
            char ch = buf[0];

            if (ch == '\n')  // 遇到换行符 → 处理完整一行
            {
                line_buf[idx] = '\0';
                if (strstr(line_buf, expected_str) != NULL)
                    return 0;          // 找到了目标字符串 → 成功
                if (strstr(line_buf, "ERROR") || strstr(line_buf, "FAIL"))
                    return -1;         // 收到错误 → 失败
                idx = 0;                // 重置，开始累积下一行
            }
            else if (ch != '\r')    // 忽略回车符
            {
                line_buf[idx++] = ch;   // 累积到 line_buf
                line_buf[idx] = '\0';

                // ★ 不等到换行就检查（为了匹配 ">" 提示符）
                if (strstr(line_buf, expected_str) != NULL)
                    return 0;
            }
        }
        osDelay(1);  // 让出 CPU 给其他任务
    }
    return -2;  // 超时返回
}

4.3 为什么 ">" 需要特殊处理？

曾遇到的 Bug：AT+CIPSEND 发送后，ESP8266 返回 >（不带换行符），原代码只在遇到 \n 时才检查匹配，永远检测不到 >，导致函数超时返回 -2。

修复方案：每个字符累加后立即检查一次（第 100-105 行）。这样 > 进入缓冲区后马上被 strstr 匹配到，不等换行。

4.4 AT 命令交互实例

AT\r\n                       AT\r\n\r\nOK\r\n
ATE0\r\n                     ATE0\r\n\r\nOK\r\n
AT+CWMODE=1\r\n              \r\nOK\r\n

AT+CIPSEND=...\r\n           \r\n>                           // 注意: 没有 \n!
GET /... HTTP/1.1\r\n...     \r\nSEND OK\r\n\r\n+IPD,410:HTTP/1.1...\r\n...\r\n
AT+CIPCLOSE\r\n              \r\nCLOSED\r\n\r\nOK\r\n

五、HTTP GET 核心流程

esp8266_http_get 是整个驱动中最重要的函数。下面分步骤拆解：

5.1 完整代码（带注释）

int esp8266_http_get(const char* host, uint16_t port,
                      const char* path,
                      char* response, uint32_t resp_len)
{
    (void)port;  // 忽略 port 参数，固定走 TCP:80

    char cmd[128];       // AT 命令缓冲区
    char http_req[300];   // HTTP 请求缓冲区
    int ret;

    /* 步骤1: 清空 response 和环形缓冲区 */
    memset(response, 0, resp_len);
    esp8266_rx_flush();

    /* 步骤2: 建立 TCP 连接 */
    snprintf(cmd, sizeof(cmd),
             "AT+CIPSTART=\"TCP\",\"%s\",80\r\n", host);
    esp8266_send_cmd(cmd);
    ret = esp8266_wait_response("OK", 10000);
    if (ret != 0) return -1;   // TCP 连接失败

    /* 步骤3: 构造 HTTP GET 请求 */
    snprintf(http_req, sizeof(http_req),
        "GET %s HTTP/1.1\r\nHost: %s\r\nConnection: close\r\n\r\n",
        path, host);

    /* 步骤4: 告诉 ESP8266 要发送多长的数据 */
    snprintf(cmd, sizeof(cmd), "AT+CIPSEND=%d\r\n", (int)strlen(http_req));
    esp8266_send_cmd(cmd);
    ret = esp8266_wait_response(">", 5000);
    if (ret != 0) return -3;   // 没收到 > 提示符

    /* 步骤5: 发送实际的 HTTP 请求 */
    esp8266_send_data(http_req, strlen(http_req));

    /* ⭐ 步骤6: 批量接收响应数据（核心部分）*/
    {
        uint32_t start = HAL_GetTick();
        uint32_t idx = 0;

        while (HAL_GetTick() - start < 15000)  // 最多等 15 秒
        {
            char batch[64];                        // 一次读 64 字节
            int n = esp8266_rx_read(batch, sizeof(batch));
            for (int i = 0; i < n && idx < resp_len - 1; i++)
                response[idx++] = batch[i];        // 追加到 response 缓冲区
            if (n == 0) osDelay(1);               // 无数据时才延迟
        }
        response[idx] = '\0';                     // 字符串结束
    }

    /* ⭐ 步骤7: 数据清洗（见下一节）*/
    // ... 过滤 +IPD 帧头、提取 JSON ...

    /* 步骤8: 关闭 TCP 连接 */
    esp8266_send_cmd("AT+CIPCLOSE\r\n");
    HAL_Delay(200);

    return (idx > 0) ? 0 : -2;  // idx>0 表示有有效数据
}

5.2 批量读取 vs 逐字节读取

对比项逐字节（改前）批量读取（改后）
每轮读取1 字节最多 64 字节
读取速率~1 字节/ms~64 字节/ms
到达速率 (115200)11.5 字节/ms11.5 字节/ms
缓冲区是否积压✓ 积压（读得慢）✗ 不会积压（读得快）
CPU 占用每字节一次 osDelay(1)=1ms无数据时才延时
为什么改前会丢数据？ 410 字节在 35ms 内突发送达，逐字节读取每轮读 1 字节 + 等 1ms = 1 字节/ms。到达速率 11.5 字节/ms，差值 10.5 字节/ms 在 35ms 内溢出：10.5 × 35 ≈ 367 字节丢失。

六、数据清洗算法（核心难点）

6.1 问题背景

ESP8266 AT 固件在收到 TCP 数据后，用 +IPD,长度: 格式包装数据：

// ESP8266 原始输出
+IPD,410:HTTP/1.1 200 OK\r\nContent-Type: application/json\r\n\r\n{"results":[{...}]}

// 如果 TCP 分片，可能变成两个 +IPD 帧
+IPD,200:HTTP/1.1...\r\n\r\n{...部分JSON...
+IPD,210:...剩余JSON...}

如果直接按大小切割，+IPD,210: 会出现在 JSON 中间，导致解析失败。

6.2 解决方案：事后过滤

// 第一步：在原始数据中找 JSON 起始位置（第一个 { 或 [）
char *json = strstr(response, "{");
if (json == NULL) json = strstr(response, "[");

// 如果在 HTTP 头里找不到，尝试在 \r\n\r\n 之后找
if (json == NULL) {
    char *rnrn = strstr(response, "\r\n\r\n");
    if (rnrn) {
        json = strstr(rnrn + 4, "{");
        if (json == NULL) json = strstr(rnrn + 4, "[");
    }
}

// 第二步：找到了 JSON → 过滤 +IPD 帧头
if (json != NULL)
{
    char *r = json, *w = json;   // r=读指针, w=写指针
    while (*r)
    {
        if (strncmp(r, "+IPD,", 5) == 0)  // 发现 +IPD 开头
        {
            r += 5;                          // 跳过 "+IPD,"
            while (*r >= '0' && *r <= '9') r++;  // 跳过数字长度
            if (*r == ':') r++;                 // 跳过冒号
            continue;   // 不复制，继续读下一个字符
        }
        *w++ = *r++;        // 正常字符：从 r 复制到 w
    }
    *w = '\0';
    memmove(response, json, w - json + 1);  // 移到缓冲区头部
}

6.3 原地过滤算法图解

原始数据:  {......JSON......}+IPD,210:......JSON......}
            ^                            ^
            r (读指针)                   w (写指针)
            俩指针都指向 JSON 起始位置

处理 "+IPD,210:" 时:
  r → 跳过 "+IPD,210:"，指向后续字符
  w → 保持不变（不写入）
            ↓
过滤后:     {......JSON...............JSON......}
            ^                      ^
            w 写到这里              r 读到这里
            ↑ 二者不会交叉，因为 r 永远 ≥ w

算法保证：读指针 r 永远 >= 写指针 w，因为读指针要么跳过（超过 w），要么和 w 同步前进。所以不会出现"写指针追上读指针"覆盖未读数据的情况。

为什么要用 memmove 而不是 memcpy？
memmove 可以处理"源和目标重叠"的情况。由于过滤后数据变短了，memmove 把过滤后的 JSON 移到 response 开头是安全的。

七、关键问题与经验教训

7.1 Bug 汇总

#问题根因修复
1 ESP8266 轮询阻塞 CPU 使用 __HAL_UART_GET_FLAG 忙等待 改用 USART1 中断 + 环形缓冲区
2 ">" 提示符检测不到 只在收到 \n 时检查匹配字符串 每个字符后都检查一次
3 环形缓冲区溢出丢数据 逐字节读取太慢, 256B 缓冲区太小 512B 缓冲区 + 批量读取 64 字节/轮
4 +IPD 分帧污染 JSON 实时过滤无法处理多帧数据 事后收集全部 + 原地过滤算法
5 NVIC 未使能导致中断不触发 HAL_UART_Receive_IT 不配置 NVIC 在 mspInit 中添加 HAL_NVIC_EnableIRQ

7.2 关键经验教训

教训1：RTOS 下的串口必须用中断或 DMA

不要用轮询！ 在裸机程序里 while(!__HAL_UART_GET_FLAG(...)) 可能勉强能用，但在 FreeRTOS 中，这会阻塞当前任务，让所有低优先级任务饿死。更重要的是，ESP8266 的 115200 baud 数据流很密集，轮询极易丢字节。

教训2：读取速率 ≥ 到达速率

这是一个经典的"生产者-消费者"问题。ESP8266 以 11.5 字节/ms 的速度发送数据，消费者（任务）的读取速率必须 >= 这个值，否则缓冲区必然溢出。批量读取是唯一正确的做法。

教训3：ESP8266 不支持多任务并发访问

ESP8266 AT 固件是单线程设计的。如果两个任务同时向 ESP8266 发 AT 命令，数据在 TX 线上交错，ESP8266 收到乱码后返回 "busy s..."。这个 bug 在开发中反复出现（NTP vs HTTP GET 的竞争），最终靠事件组分阶段同步解决。

教训4：AT 命令的 15 秒超时浪费

早期代码先尝试 SSL:443（15 秒超时），失败后再回退到 TCP:80。但 ESP8266-01S 的 AT 固件不支持 SSL，每次白等 15 秒。简化后的代码直接走 TCP:80。
