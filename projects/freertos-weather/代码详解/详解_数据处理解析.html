天气数据 JSON 解析模块详解

从零理解：手写 JSON 解析器、防御性编程、ASCII 过滤、空指针保护

一、模块概述

weather.c 负责将 ESP8266 获取的 HTTP 响应（JSON 格式）解析为结构化的天气数据。核心只有：

  • 1 个结构体：WeatherData_t 用于存储解析结果
  • 2 个函数：json_find_value（查找字段）+ weather_parse_now（完整解析）
  • 1 个辅助函数：sanitize_to_ascii（过滤中文字符）
核心设计思想：没有使用 cJSON 等第三方 JSON 库，而是手写了一个极简的 JSON 解析器。原因：（1）STM32F103C8 只有 20KB RAM，cJSON 的动态内存分配会碎片化堆；（2）API 返回格式固定，用 strstr 定位字段效率更高；（3）代码量可控（整个 weather.c 仅 124 行）。

二、数据结构 — WeatherData_t

// weather.h:10-15
typedef struct {
    char city[32];       // 城市名，如 "Hangzhou"
    char weather[32];    // 天气描述，如 "Cloudy"
    char temp[8];        // 温度，如 "27"（不带单位）
    char humidity[8];    // 湿度，如 "65" 或 "--"
} WeatherData_t;
字段最大长度数据来源说明
city32 字节location → name国际城市名（英文）
weather32 字节now → text天气描述（英文）
temp8 字节now → temperature纯数字字符串
humidity8 字节now → humidity免费版 API 不返回，始终 "--"

为什么都用 char 数组而不是 int？

  • JSON 解析直接得到的是字符串（"27" 而不是整数 27）
  • OLED 显示需要字符串格式，避免整数转字符串的转换开销
  • "--" 这样的占位符只能用字符串表示

3.2 解析目标

JSON 路径目标字段

results → [0] → now → textweather.weather = "Cloudy"
results → [0] → now → temperatureweather.temp = "27"

四、json_find_value 函数逐行拆解

这是整个模块最底层的函数，负责在一个 JSON 片段中搜索指定 key 并提取它的值。

static char* json_find_value(const char* json,
                              const char* key,
                              char* value, int value_len)
{
    // 步骤1: 在 JSON 中搜索 key 字符串
    char* p = strstr(json, key);
    if (p == NULL) return NULL;     // 找不到 key

    // 步骤2: 跳过分隔符，找到冒号
    // 输入: ..."key":"value"...  → p 指向 key
    p = strchr(p, ':');            // 找冒号
    if (p == NULL) return NULL;
    p++;                             // 跳过冒号

    // 步骤3: 跳过空格和制表符
    while (*p == ' ' || *p == '\t') p++;

    // 步骤4: 判断值类型
    if (*p == '"')                 // 字符串值: "Cloudy"
    {
        p++;                         // 跳过开头引号
        char* end = strchr(p, '"');  // 找结尾引号
        if (end == NULL) return NULL;

        int len = end - p;          // 计算字符长度
        if (len > value_len - 1)     // 防止缓冲区溢出
            len = value_len - 1;
        strncpy(value, p, len);      // 复制字符串
        value[len] = '\0';          // 手动加结束符
        return value;
    }
    else if (*p == '{' || *p == '[')  // 对象/数组 → 不支持
    {
        return NULL;
    }
    else                             // 数字值: 27  (不带引号)
    {
        char* end = p;
        while (*end != ',' && *end != '}' &&
               *end != '\n' && *end != '\r')
            end++;                    // 找到值的结尾

        int len = end - p;
        if (len > value_len - 1) len = value_len - 1;
        strncpy(value, p, len);
        value[len] = '\0';
        return value;
    }
}

4.1 处理示例

输入key输出
"name":"Hangzhou""name""Hangzhou"（走字符串分支）
"temperature":"27""temperature""27"（走字符串分支）
"humidity":"65""humidity""65"（走字符串分支）
"now":{...}"now"NULL（走对象分支，不支持提取对象值）

4.2 关键边界处理

  • strstr 的模糊匹配：搜索 "name" 可能会匹配到 "name_cn"。但心知天气 API 返回的 key 名称是固定的，不会混淆。如果担心可以加 strstr(p, "\"name\"") 精确匹配。
  • 缓冲区溢出保护：如果值长度超过 value_len，截断处理（第 22、39 行的 len 检查）
  • strncpy 不自动加 '\0'：如果 len == value_len，strncpy 不会写结束符，需要手动加（第 24、41 行）
曾踩过的坑：strstr 在搜索小 key 时可能匹配到大 key 的子串。例：在 JSON 中搜索 "now" 时，如果前一个字段名包含 "now"（如 "know"），会误匹配。心知天气 API 的字段名中没有这种命名冲突，所以这个方法可行。如果遇到更复杂的 JSON，还是应该用 cJSON 等专业库。

五、weather_parse_now 完整流程

这个函数是解析的入口，它将 HTTP 响应字符串按 JSON 层次结构逐个字段提取。

int weather_parse_now(const char* json_response, WeatherData_t* weather)
{
    /* 阶段1: 参数安全检查 */
    if (json_response == NULL || weather == NULL)
        return -1;

    memset(weather, 0, sizeof(WeatherData_t));  // 初始化为0

    /* 阶段2: 定位 JSON 层次结构 */
    const char* results_start = strstr(json_response, "results");
    if (results_start == NULL) return -2;

    const char* first_brace = strchr(results_start, '[');   // 到 results 数组
    if (first_brace == NULL) return -3;

    const char* first_obj = strchr(first_brace, '{');       // 到第一个对象
    if (first_obj == NULL) return -4;

    /* 阶段3: 提取 location.name → city */
    const char* location_start = strstr(first_obj, "location");
    if (location_start != NULL)
        json_find_value(location_start, "name", weather->city, sizeof(weather->city));

    /* 阶段4: 提取 now.text/temperature/humidity */
    const char* now_start = strstr(first_obj, "now");
    if (now_start != NULL)
    {
        json_find_value(now_start, "text", weather->weather, sizeof(weather->weather));
        json_find_value(now_start, "temperature", weather->temp, sizeof(weather->temp));
        json_find_value(now_start, "humidity", weather->humidity, sizeof(weather->humidity));
    }

    /* 阶段5: 数据清洗和 Fallback */
    sanitize_to_ascii(weather->city);
    sanitize_to_ascii(weather->weather);
    sanitize_to_ascii(weather->temp);
    sanitize_to_ascii(weather->humidity);

    if (strlen(weather->city) == 0)     strcpy(weather->city, "Unknown");
    if (strlen(weather->weather) == 0)  strcpy(weather->weather, "Unknown");
    if (strlen(weather->temp) == 0)     strcpy(weather->temp, "--");
    if (strlen(weather->humidity) == 0) strcpy(weather->humidity, "--");

    return 0;  // 成功
}

5.1 逐阶段解析图解

1 找 "results" —— 从原始 JSON 中定位 results 数组的起始位置
strstr(json, "results") → 指向 "...results":[{...}]" 中 "r" 的位置
2 找 '[' —— 从 results_start 开始找数组开始的 '[' 字符
strchr(results_start, '[') → 指向 "[{...}]" 中 '['
3 找 '{' —— 从 first_brace 开始找第一个对象的 '{'
strchr(first_brace, '{') → 指向 "{...}" 中 '{'
4 找 "location" → 调用 json_find_value 提取 name
5 找 "now" → 依次提取 text / temperature / humidity

5.2 为什么用 char* 指针缩小搜索范围？

当定位到 first_obj（指向第一个 {）后，后面的 strstr(first_obj, "location") 只在 { 之后搜索。这相当于"以当前对象为根"做搜索，避免了匹配到前面无关内容。虽然 strstr 是全字符串搜索，但因为作用范围缩小了，可以安全地提取字段。

更深刻的思考：为什么不用 strstr(json, "location")（从开头搜）？因为 JSON 中可能有多个 location 字段（比如天气预报里，每一天都有 location），从 first_obj 开始搜确保找到的是当前对象的 location。

六、ASCII 过滤 — sanitize_to_ascii

6.1 为什么需要这个函数？

心知天气 API 可以根据 language=en 参数返回英文结果。但如果 API 配置有误或 fallback 到中文，OLED 显示中文会变成空白——因为 oled_show_char 只支持 ASCII 字符（0x20~0x7E），遇到 >= 0x80 的 UTF-8 中文字节直接跳过。

更隐蔽的问题是：一个中文字符占 3 个 UTF-8 字节，每个字节都 >= 0x80。如果全部被过滤，字符串长度变为 0，之前的代码不会触发 Fallback 显示 "Unknown"，因为 strlen("深圳") = 6 > 0。

6.2 代码

static void sanitize_to_ascii(char* str)
{
    char* r = str;     // 读指针
    char* w = str;     // 写指针（原地操作）
    while (*r)
    {
        if ((unsigned char)*r < 0x80)  // ASCII 字符保留
            *w++ = *r;
        // >= 0x80 的非 ASCII 字符跳过（不写）
        r++;
    }
    *w = '\0';  // 新结束符
}

6.3 举例

输入过滤后strlen 过滤前strlen 过滤后
"Hangzhou""Hangzhou"88
"深圳"（UTF-8: E6 B7 B1 E5 9C B3）""（空）60 ← 触发 Fallback！
"Cloudy 多云""Cloudy "117
与原代码配合：过滤后如果为空字符串，后面的 Fallback 逻辑将其替换为 "Unknown" 或 "--"，确保 OLED 始终有内容显示。

七、关键问题与经验教训

7.1 曾发生的 Bug 汇总

#问题根因修复
Bug 1 HardFault (死机黑屏) strchr 返回 NULL 后未检查，直接传给下一个 strchr，触发空指针解引用 添加 NULL 检查，返回不同错误码 -3/-4

7.2 Bug 1 详细分析：first_brace 空指针

现象：OLED 显示 "Weather OK!" 后突然黑屏，全黑无任何像素。
实际原因：HardFault 导致系统复位，OLED 重新初始化（清屏）。 
// 改前（weather.c 早期版本）
const char* first_brace = strchr(results_start, '[');
const char* first_obj = strchr(first_brace, '{');   // ← first_brace 可能是 NULL！
if (first_obj == NULL) return -3;

// 改后
const char* first_brace = strchr(results_start, '[');
if (first_brace == NULL) return -3;  // ← 先检查 first_brace
const char* first_obj = strchr(first_brace, '{');
if (first_obj == NULL) return -4;  // ← 再检查 first_obj，返回不同错误码

为什么 results 可能不是数组？ 心知天气 API v3 的 results 通常是数组 "results":[{...}]，但在 API 异常时可能返回 "results":{...}（对象而非数组），或者干脆错误信息中根本没 results。

7.4 经验教训总结

教训1：NULL 指针不止检查参数

strstr/strchr 的返回值也是潜在的 NULL 源！对外部数据（网络响应）的解析要假设格式可能不符合预期。每步解析都做 NULL 检查是嵌入式开发的铁律。

教训2：外部数据不可信

API 返回的数据可能因网络截断、SSL 解密错误、API 服务升级、key 过期等原因不符合预期。每条解析路径都要有防御性编程保护。

教训3：不要信任第三方 API 的"免费版"

心知天气免费版不返回湿度字段。这不是 bug，但设计时需要提前确认 API 文档。更稳妥的做法是设计时假设"所有字段都可能缺失"，并用 Fallback 值兜底。

教训4：不同的错误码帮助调试

使用 -1、-2、-3、-4 四个不同错误码，比统一返回 -1 更有价值。在 OLED 上显示 "Err=-3" 可以直接定位到"没找到 '['" 的问题。
